# <a name="party-and-global-address-book"></a>当事者およびグローバル アドレス帳

[!include [banner](../../includes/banner.md)]

[!include [rename-banner](~/includes/cc-data-platform-banner.md)]

当事者アドレス帳およびグローバル アドレス帳は、Finance and Operations アプリケーション の概念です。 当事者は、組織または個人にできます。 **当事者** のプロパティ (名前、言語、連絡先、住所など) をグローバルに保存および管理することができます。 1 つの場所でプロパティ値が変更された場合、**当事者** が関係するすべての場所にプロパティ値が反映されます。

## <a name="party"></a>当事者

*当事者* とは 、業務に関する人物または組織です。 当事者の概念を使用すると、ある人または組織が 1 つの企業において複数の役割 (作業者、顧客、仕入先、または連絡先) を担うことが可能です。 ロールは、コンテキストおよび目的に基づいて設定されます。 Contoso と Fabrikam という 2 つの架空の会社の例を次に示します。

+ **作業者**: 従業員。 たとえば、Contoso の従業員です。
+ **仕入先**: 仕入先組織、または会社に商品やサービスを提供している唯一の個人事業主。 たとえば、Fabrikam が Contoso に供給品を販売している場合、Fabrikam は仕入先の役割を果たしていることになります。
+ **連絡先**: 連絡する対象の担当者。 たとえば、Contoso が Fabrikam から供給品を購入した場合、Contoso の従業員が Fabrikam の連絡先に連絡します。
+ **顧客**: 顧客は、会社から物品を購入する担当者または会社です。 たとえば、Contoso が Fabrikam から供給品を購入した場合、Contoso は Fabrikam の顧客となります。

当事者モデルは、多くの場合、当事者が複数の役割を持つ場合など、組織と人との中規模なリレーションシップから複雑なリレーションシップを表するために使用されます。 次にいくつか一般的な例を挙げます。

+ 当事者は、顧客と仕入先両方の可能性があります。 たとえば、北アメリカでは、Fabrikam が Contoso に対して電線を販売し、Contoso の組み立て式スピーカーを購入しました。 ヨーロッパでは、Fabrikam は Contoso に部品を販売しますが、Contoso の製品は購入しません。
+ 当事者は、従業員と顧客両方の可能性があります。 たとえば、Contoso の従業員が Contoso から電子製品を個人的に購入したとします。
+ 人物と組織の間には多対多のリレーションシップがある可能性があります。 たとえば、Fabrikam はサービスのスペシャリストを供給し、配置コーディネーターを採用しています。 コーディネーターは、サービスのスペシャリストに、Fabrikam の複数の顧客からの作業依頼に対応します。 Contoso は顧客アカウントの 1 つです。 Contoso にスペシャリストを必要な場合は、コーディネーターに連絡してその要求に対応してもらいます。 コーディネーターは、すべての顧客の要求を処理し、多対多のリレーションシップを作成します。

次の画像は、当事者のデータ モデルを示しています。

![当事者のデータ モデル](media/party-gab-image1.png)

> [!Tip]
> 新しいアカウント レコードを作成する場合は、[当事者] フィールドを使用してそのレコードを名前で検索します。 レコードが見つかったら、そのレコードを選択する必要があります。 当事者からのデータはすべて自動的に入力されます。 すべての必須フィールドを手動で入力する必要はありません。 この動作は、最初から組み込まれている [アカウント]、[連絡先]、および [仕入先] フォームに入っています。

Finance and Operations アプリのすべての当事者ロールが二重書き込みでサポートされているわけではありません。 当事者ロールの完全なリストについては、[グローバル アドレス帳の概要](../../../fin-ops/organization-administration/overview-global-address-book.md)を参照してください。

### <a name="global-address-book"></a>グローバル アドレス帳

グローバル アドレス帳は、事業に参加する組織および個人の住所および電子住所のディレクトリです。

グローバル アドレス帳では、必要な数の住所および電子住所が格納および処理されます。 たとえば、Fabrikam には 50 か所にガソリンスタンドがあるとします。 場所ごとに異なる住所、電子住所、および電話番号があります。 すべての業務関連の購入はメインのガソリン スタンドで請求されますが、購入は、購入を要求した特定のガソリン スタンドまで直接出荷されます。 グローバル アドレス帳には、請求先住所としてメインのガソリン スタンドを格納し、Fabrikam の出荷先住所として各ガソリン スタンドを指定します。 住所は一度格納し、必要に応じて見積書や注文で取得できます。

1 人または組織が業務コンテキストに基づいて複数の役割を果たすことができます。 その場合、住所と電子住所が同じ場合があります。 この場合、あるロールの住所の変更がもう一方のロールに表示され、逆も起こります。 グローバル アドレス帳は、住所をグローバルに格納して処理します。

## <a name="contacts"></a>連絡先

customer engagement アプリケーションでは、*連絡先* は人物です。 ただし、**連絡先** テーブルは、人物、ポータル ユーザー、B2C 顧客、または仕入先を表すために過負荷状態となっています。 表現は暗黙的であり、関連するトランザクションを調査しないとその差は見分けられません。 **連絡先** テーブルは **アカウント** テーブルとの 1:1 のリレーションシップに制限されています。 当事者およびグローバル アドレス帳モデルの一部として、二重書き込みでは、分類および二重書き込み用の明示的なプロパティが導入され、**連絡** 担当者と組織 (アカウント エンティティまたは仕入先エンティティ) 間の N:N のリレーションシップが可能になります。

**連絡先** 行には 2 種類あります。

+ ストライプ連絡先: **会社** フィールドに必須値が設定された連絡先行。
+ 非ストライプ連絡先: 空白の **会社** フィールドが空白の連絡先行。

**連絡先** テーブルには、次のタイプの行を格納できます。

行タイプ | 説明
---|---
販売可能な連絡先や B2C 顧客など、顧客の担当者。 | **会社** フィールドが空白ではなく、**顧客である** フィールドが **はい** に設定されている、ストライプ連絡先レコード。
仕入先である個人。たとえば、仕入先のような単独の販売者です。 | **会社** フィールドが空白ではなく、**仕入先である** フィールドが **はい** に設定されている、ストライプ連絡先レコード。
顧客と仕入先両方である人物。 | **会社** フィールドが空白ではなく、**顧客である** フィールドが **はい** に設定されており、**仕入先である** フィールドが **はい** に設定された、ストライプ連絡先レコード。 ある人物を、1 つの製品の生産者と別の製品の消費者の両方に指定できます。 Finance and Operations アプリと二重書き込みの両方でこのリレーションシップがサポートされます。
組織の連絡担当者で、顧客でも仕入先でもない担当者。 | **会社** フィールドが空白ではなく、**顧客である** フィールドが **いいえ** に設定されており、**仕入先である** フィールドが **いいえ** に設定された、非ストライプ連絡先レコード。

## <a name="contact-for-party"></a>当事者の連絡先

**当事者の連絡先** は、**アカウント** 行と **連絡先** 行間の N:N のリレーションシップを補完して処理します。 非ストライプ行からストライプ **連絡先** 行を除外して、非ストライプ **連絡先** 行のみを **アカウント** または **仕入先** 行に関連付けることができます。

たとえば、Natasha Jones と Miguel Reyes は、担当地域内の農場を診察する獣医です。 Natasha はシアトル、Miguel はケント地域を担当しています。 customer engagement アプリケーションでは、農場が顧客として表され、獣医は連絡担当者です。 Natasha の単一 **連絡先** レコードは、Natasha がサービスを提供しているすべての農場と関連付けられています。 同様に、Miguel の単一 **連絡先** レコードは、Miguel がサービスを提供しているすべての農場と関連付けられています。

これらのリレーションシップは、**当事者の連絡先** テーブルに格納されます。 この情報は、すでに組み込まれているフォームに表示されています。

+ **アカウント** フォームを使用すると、**関連付けられた連絡先** という名前のタブが表示されます。 1 つ以上の連絡先を **アカウント** 行に関連付けるには、このタブを使用 します。 このフォームでは、組織の連絡担当者を割り当てる必要があります。 連絡先を割り当てたら、そのアカウントの基本連絡先として連絡先を選択できます。 **簡易作成** フォームを使うと、連絡担当者のみを選択できます。 **仕入先** フォームを使用している場合の動作は同じで、レコード タイプが **組織** です。
+ **連絡先** フォームで、行が顧客か仕入先か、あるいはその両方 (関連付けられた連絡先) の場合、**関連付けられた連絡先** というタブがあります。 1 つ以上の連絡先を関連付けるには、このタブを使用 します。 このフォームでは、B2C 顧客または仕入先の連絡担当者を割り当てています。 連絡先を割り当てたら、そのアカウントの基本連絡先として連絡先を選択できます。 **簡易作成** フォームを使うと、連絡担当者のみを選択できます。
+ **連絡先** フォームで、行が連絡担当者 (非ストライプ連絡先) の場合、**関連付けられた組織** というタブがあります。 1 つ以上の顧客または仕入先を関連付けるには、このタブを使用します。 このフォームでは、顧客または仕入先を基本の連絡担当者に割り当てています。 顧客または仕入先は、組織、人物、またはその両方を設定できです。 随時 4 つのフィールドから 1 つの値のみを選択できます。

    ![[関連付けられている組織] タブ](media/party-gab-image3.png)

    + **当事者 ID** を選択した場合、基本の連絡先は、選択した当事者のすべてのロールに割り当てられます。
    + **関連付けられた連絡先** を選択した場合、タイプが人物であるストライプ連絡先を選択しています。
    + **関連付けられた勘定** または **仕入先** を選択した場合、組織を選択します。

    選択に関係なく、関連は当事者レベルで作成され、当事者のすべてのロールに適用され、[当事者への連絡] エンティティに格納されます。

> [!Note]
> customer engagement アプリケーションの **当事者の連絡先** テーブルの表示名は、**顧客/仕入先の連絡先** です。

**顧客である** が **いいえ** で **仕入先である** が **いいえ** である **連絡先** 行を開くと、**関連付けられた組織** タブが表示されます。このタブを使用して、1 つ以上の顧客または仕入先の組織を連絡先に関連付けます。

**顧客である** が **はい** で **仕入先である** が **はい** である **連絡先** 行を開くと、**関連付けられた連絡先** タブが表示されます。このタブを使用して、1 つ以上の連絡先を関連付けます。

## <a name="postal-address"></a>住所

**アカウント**、**連絡先**、および **仕入先** フォームに、**住所** という名前の新しいタブが表示されます。 この画像に示すように、グリッドを使用したサポート N 住所用の **住所**。

![住所のグリッド](media/party-gab-image4.png)

+ **住所のロール** 列には、住所の目的が一覧表示されます。
+ **プライマリである** 列には、プライマリ住所が一覧表示されます。
+ **住所番号** 列には、住所の順序が一覧表示されます。
+ **+新しい住所** ボタンを使用すると、新しい住所を作成できます。 必要な数の住所を作成することができます。

**アカウント** フォームの **集計** タブにある **住所 1** フィールドと **住所 2** フィールドが、それぞれ **配送先** 住所と **請求先** 住所 に対応します。

![住所の集計タブ](media/party-gab-image5.png)

**連絡先** フォームの **集計** タブにある **住所 1** フィールド、**住所 2** フィールド、および **住所 3** フィールドが、それぞれ **ビジネス** 住所、**配送先** 住所、および **請求先** 住所に対応します。

## <a name="electronic-address"></a>電子アドレス

**アカウント**、**連絡先**、および **仕入先** フォームに、**電子住所** という名前の新しいタブが表示されます。 この画像に示すように、グリッドを使用したサポート N 住所用の **住所**。

![電子住所のグリッド](media/party-gab-image6.png)

+ **タイプ** 列には、住所のタイプが一覧表示されます。
+ **プライマリである** 列には、プライマリ住所が一覧表示されます。
+ **目的** 列には、電子住所の目的が一覧表示されます。
+ **+新しい電子住所** を使用すると、新しい住所を作成できます。 必要な数の住所を作成することができます。

電子住所は、このグリッドでのみ使用できます。 将来のリリースでは、**集計** タブや **詳細** タブなど、他のタブからすべての電子住所フィールドと郵便番号フィールドが削除されます。

1. 以下のマップは、もう不要なので停止します。 代わりに、Contacts V2 (msdyn_contactforparties) マップを実行します。

    + CDS 連絡先 V2 および連絡先 (顧客連絡先を参照)
    + CDS 連絡先 V2 および連絡先 (仕入先連絡先を参照)

2. 次のエンティティ マッピングが当事者機能に対して更新されたため、これらのマッピングには最新バージョンを適用する必要があります。
