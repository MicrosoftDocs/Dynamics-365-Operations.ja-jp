# <a name="manage-business-partner-users-on-b2b-e-commerce-websites"></a>B2B eコマース Web サイトでのビジネス パートナー ユーザーの管理

[!include [banner](../../includes/banner.md)]

この記事では、Microsoft Dynamics 365 Commerce 企業間 (B2B) eコマース Web サイトと Commerce headquarters でビジネス パートナー ユーザーを追加、編集、削除する方法について説明します。

> [!NOTE]
> - このドキュメントには、[顧客階層を使用して B2B ビジネス パートナーを管理する](partners-customer-hierarchies.md) 記事が前提条件となります。
> - **組織管理 \> ドキュメント管理 \> ドキュメント タイプ** で **ドキュメント タイプ** を開き、コマース本部のドキュメント タイプ エンティティを初期化します。

B2B e コマースの Web サイトでは、組織が取引相手となるために登録する必要があります。 組織が B2B e コマースの Web サイトに登録の詳細を送信すると、登録要求は資格審査プロセスを経由します。 組織が審査つ通貨した場合は、ビジネスパートナーとしてオンボードされます。

組織がビジネス パートナーとしてオンボードされた後、ビジネスパートナーへの要求を開始した組織ユーザーは、管理者ユーザーとして識別され、B2B e コマース Web サイトの承認された追加のユーザーに権限を与える特権が付与されます。 これらの認可されたユーザーは、ビジネスパートナーの代理として注文を出すことができます。

## <a name="set-up-the-administrator-user-for-a-new-business-partner"></a>新しいビジネス パートナーで使用する管理者ユーザーの設定

潜在的なビジネス パートナーは、B2B Web サイト上のリンクを介してオンボーディング リクエストを送信することで、B2B e コマース Web サイトへのオンボーディング プロセスを開始することができます。 カスタマイズ可能なフォームを使用して、変更や登録に必要な詳細を入力できます。 要求が送信されると、送信確認ページが表示されます。 申請が承認された場合、ビジネス パートナーになるための要求を提出した変更の要求を開始した企業と要求者 (オンボード要求を発動したユーザー) がビジネス パートナーの管理者ユーザーになります。

Commerce 本部のビジネス パートナーの要求を承認するには、以下の手順に従ってください。

1. **小売とコマース IT \> 配送スケジュール** の順に移動します。
1. **P-0001** ジョブを実行して、すべてのビジネス パートナーが要求を Commerce 本部に取り込みます。
1. **P-0001** ジョブが正常に実行された後、**小売とコマース IT \> 顧客** に移動し、**顧客とチャンネル要求を同期する** ジョブを実行します。 このジョブが正常に実行された後、オンボードの要求は、Commerce 本部の **B2B 見込み顧客** タイプの見込顧客レコードとして作成されます。 
1. **顧客 \> すべての見込顧客** に移動して、新しいビジネス パートナーの見込顧客レコードを選択して、見込顧客の詳細ページを開きます。
1. **全般** タブ で、**変換 \> 承認/否認** を選択し、オンボード要求を承認または否認します。 確認メッセージが表示されたら、処理の続行を確認し、要求を承認します。 要求は、見込顧客レコードの **状態** フィールドが **承認済** に変更されます。 申請者のメールアドレスにメールが送信され、組織がビジネスパートナーとして承認されたことを確認します。 顧客階層も作成され、要求者がビジネス パートナーの管理者として追加されます。

    > [!NOTE]
    > 現在、確認メールは承認時にすぐに送信されます。 ただし、将来の Commerce 機能を使用すると、管理者が電子メールを手動でトリガできます。

1. **小売とコマース IT \> 配送スケジュール** に移動し、**1010 (顧客)** ジョブを実行し、新規作際された顧客と顧客の階層レコードをチャネル データベースに プッシュします。

> [!NOTE]
> 新しい顧客レコードをチャンネル データベースに確実に送信するには、その顧客に関連付けられているアドレス帳を、オンライン店舗に関連付けられている顧客アドレス帳に含める必要があります。 このプロセスを自動化するには、オンライン ストアの既定の顧客のアドレス帳を構成して、すべての新しい顧客にアドレス帳の値がコピーされます。

顧客階層レコードがチャネル データベースに同期されると、要求者は、リクエストを送信した際に提供されたメールアドレスを使用して、B2B e コマース Web サイトにサインインすることができます。 ユーザーは、登録フローを使用してアカウントのパスワードを定義できます。 見込顧客の承認で作成された B2C ID プロバイダー レコードを B2B 顧客レコードにリンクするために Azure Active Directory (Azure AD) B2C ID プロバイダー レコードを有効にする方法の詳細については、[自動リンクの有効化](../identity-record-linking.md) を参照してください。

## <a name="notify-b2b-prospects-when-they-are-approved-or-rejected"></a>承認または却下された場合に B2B 見込顧客に通知する

B2B 見込顧客のオンボード要求を承認または却下すると、見込顧客にメール通知を自動的に送信できます。

**B2B 見込顧客の承認** 通知タイプまたは **B2B 見込顧客の却下** 通知タイプのイベントについて、Commerce 本部 でメール通知を設定するには、次の手順に従います。

1. **B2B 見込顧客の承認** 通知タイプまたは **B2B 見込顧客の却下** 通知タイプがトリガーされた場合に、見込顧客に送信されるメールのメール テンプレートを作成します。 これらの通知タイプがサポートするプレースホルダーについては、[通知タイプ](../email-templates-transactions.md#notification-types) を参照してください。 メール テンプレートの作成方法については、[メール テンプレートを作成する](../email-templates-transactions.md#create-an-email-template) を参照してください。
1. **B2B 見込顧客の承認** 通知タイプおよび **B2B 見込顧客の却下** 通知タイプをメール通知プロファイルに追加し、その後作成したメール テンプレートにマップします。 通知プロファイルについては、[メール通知プロファイルの設定](../email-notification-profiles.md) を参照してください。

## <a name="onboard-additional-business-partner-users"></a>追加の取引先ユーザーをオンボードする

取引先の管理者ユーザーは、必要に応じて、B2B eコマースの Web サイトに取引先ユーザーを追加で提供することができます。

追加のビジネス パートナー ユーザーを B2B e コマース Webサイトに変更するには、次の手順に従います。

1. B2B e コマースの Web サイトに管理者としてログインします。
1. **自分のアカウント \> 組織のユーザー \> 詳細の表示** に移動し、**ユーザーの追加** を選択します。
1. 必要な情報を入力し、**保存** を選択します。 新規ユーザーの状態を **保留中** に設定します。

**P-0001** および **顧客およびチャンネル要求を同期する** ジョブが実行された後は、Commerce 本部に新規ユーザーの **個人** タイプの顧客レコードが作成されます。 この顧客レコードは、関連する取引先の顧客階層レコードにも関連付けられます。 さらに、新規ユーザーのメールアドレスにメールが送信され、取引先組織のユーザーとして追加され、B2B e Web サイトにサインインできるようになったことが通知されます。

次に、**1010 (顧客)** ジョブの同期を実行して 、新しいビジネス パートナー ユーザーをチャンネル データベースに同期します。

顧客レコードが同期された後、B2B e コマース Web サイトのユーザーの状態が **有効** に設定され、新しいユーザーはメール アドレスを使用して B2B e コマース Web サイトにログインできます。 ユーザーは、登録フローを使用してアカウントのパスワードを定義できます。 Commerce 本部で作成された B2B 顧客レコードにリンクするために見込顧客の承認で作成された Azure AD B2C ID プロバイダー レコードを有効にする方法の詳細は、[自動リンクを有効にする](/dynamics365/commerce/identity-record-linking.md) を参照してください。

## <a name="edit-business-partner-user-details"></a>取引先のユーザー詳細情報を編集する

取引先ユーザーの詳細を編集するには、次の手順に従います。

1. B2B e コマースの Web サイトに管理者としてログインします。
1. **自分のアカウント \> 組織ユーザー \> 詳細の表示** に移動し、**編集** ボタン (鉛筆アイコン) を選択して必要な変更を行い、 **保存** を選択します。 この変更が反映されるのは、**P-0001**、**顧客およびチャンネル要求を同期する**、**1010 (顧客)** ジョブが実行された後にのみです。

## <a name="remove-a-business-partner-user"></a>取引先ユーザーの削除

必要に応じて、管理者は、B2B e コマース Web サイトにアクセスできるユーザーのリストから、ビジネスパートナー組織の既存のユーザーを削除することができます。
取引先ユーザーを削除するには、次の手順に従います。
- B2B e コマースの Web サイトに管理者としてログインします。
- **マイ アカウント > 組織ユーザー \> 詳細の表示** に移動して、**削除** ボタン ("X" マーク) を選択します。 確認メッセージが表示されたら、ユーザーを削除してください。 この変更が反映されるのは、**P-0001**、**顧客およびチャンネル要求を同期する**、**1010 (顧客)** ジョブが実行された後にのみです。

> [!NOTE]
> B2B eコマース Web サイトにアクセスできるユーザーの一覧からユーザーを削除すると、対応する顧客レコードがビジネス パートナーの顧客階層レコードから削除されます。 ただし、顧客レコード自体は Commerce 本部からは削除されません。

## <a name="onboard-existing-customers-as-business-partners-on-the-b2b-e-commerce-website"></a>B2B e Commerce Webサイトで既存の顧客をビジネス パートナーとしてオンボードする

管理者は、取引相手やユーザーを Commerce 本部で直接変更することができます。 この機能は、B2B e コマースの Web サイトで既存の取引相手に出張する場合に役立ちます。

Commerce 本部の取引先とユーザーをオンボードするには、以下の手順に従ってください。

1. ビジネス パートナーとして追加する、**組織** タイプの顧客を作成または選択します。
1. ビジネス パートナーの管理者またはユーザーとして追加する **個人** タイプの顧客を作成または選択します。 その基本メール アドレスは顧客と関連していることを確認してください。 これらの電子メール アドレスは、Web サイトへのログインに使用されます。 

    > [!NOTE]
    > Web サイトにログインできるユーザーの一意の顧客レコードを見つける必要があります。 法人で同じ基本電子メール アドレスを持つ顧客が複数見つける場合、そのユーザーは Web サイトにログインできます。

1. 顧客の階層 ID を作成します。
1. **名前** フィールドに、名前を入力します。
1. **組織** フィールドで、取引先組織の顧客を入力します。
1. **階層** クイックタブで、**追加** を選択します。
1. **名前** フィールドに、**個人** タイプの顧客を選択します。
1. 管理者として指定する顧客の **管理者** ロールを選択します。
1. このプロセスを繰り返し、階層にもっと顧客を追加していきます。

## <a name="additional-information"></a>追加情報

- この記事で説明されているジョブはすべて、バッチ形式でスケジュール上で実行するように構成することができます。 取引先が必要に応じてバッチジョブを構成することが意図されています。
- 現在、管理者ユーザーとして指定できるユーザー/顧客レコードは1つのみであり、そのロールは Commerce 本部でのみ変更できます。 取引先が複数の管理者を指定したり、B2B の e コマース Web サイトから管理者を変更したりできるような、セルフサービス機能には対応していません。
- ユーザーに対して支出制限を定義することができますが、注文入力プロセス中に支出制限を実施する機能はまだ実装されていません。
- B2B e コマース Web サイトでのユーザーエクスペリエンスのビジネス ロジックと検証は、Commerce 本部でユーザーにマッピングされた顧客レコードの構成に基づいています。
