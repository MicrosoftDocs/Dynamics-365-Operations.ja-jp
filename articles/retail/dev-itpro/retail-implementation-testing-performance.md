## <a name="user-acceptance-testing"></a><span data-ttu-id="bd277-105">ユーザー受け入れテスト</span><span class="sxs-lookup"><span data-stu-id="bd277-105">User acceptance testing</span></span>

<span data-ttu-id="bd277-106">ユーザー受け入れテスト (UAT) の主な目的は、特定のビジネス シナリオが期待どおりに動作するかを確認することです。</span><span class="sxs-lookup"><span data-stu-id="bd277-106">The main purpose of user acceptance testing (UAT) is to verify that specific business scenarios work as you expect.</span></span> <span data-ttu-id="bd277-107">テストはカスタマイズだけでなく、最初から用意されている Microsoft の機能と、non-happy-path テストを含める必要があります。</span><span class="sxs-lookup"><span data-stu-id="bd277-107">The testing should include not only the customizations, but also out-of-box Microsoft functionality and non-happy-path testing.</span></span> <span data-ttu-id="bd277-108">実稼働環境に移動する前に正しく動作しないものは、すべてキャッチすることが目標です。</span><span class="sxs-lookup"><span data-stu-id="bd277-108">The goal is to catch anything that doesn't work correctly before you go to the production environment.</span></span>

<span data-ttu-id="bd277-109">最良の結果を得るには、UAT 環境が、開発環境 (階層 1) ではなく、階層 2 〜階層 5 環境である必要があります。</span><span class="sxs-lookup"><span data-stu-id="bd277-109">For the best results, the UAT environment should be a Tier 2 through Tier 5 environment, not a development environment (Tier 1).</span></span>

<span data-ttu-id="bd277-110">開発環境が使用される場合、開発者がエラーを引き起こす可能性がある同じ環境を使用するシナリオがあります (例えば、未確定のソース変更またはデバッガーの添付エラーなど)。</span><span class="sxs-lookup"><span data-stu-id="bd277-110">If a development environment is used, there are scenarios where a developer who uses the same environment can cause errors (for example, uncommitted source changes or debugger attached errors).</span></span> <span data-ttu-id="bd277-111">Microsoft Internet Information Services (IIS) Express と IIS の間のスイッチは、問題を引き起こす可能性もあります。</span><span class="sxs-lookup"><span data-stu-id="bd277-111">The switch between Microsoft Internet Information Services (IIS) Express and IIS can also cause issues.</span></span> <span data-ttu-id="bd277-112">さらに、開発マシンの何が起こったかを正確に知る方法がないため、レベル 1 環境での Microsoft のサポートは非常に限られたものとなります。</span><span class="sxs-lookup"><span data-stu-id="bd277-112">Additionally, because there is no way to know exactly what has happened on a development machine, Microsoft support for a Tier 1 environment is very limited.</span></span>

<span data-ttu-id="bd277-113">UAT (たとえば、「ドライ ラン」の準備) で使用できる運用環境。</span><span class="sxs-lookup"><span data-stu-id="bd277-113">A production environment can be used for UAT (for example, as a "dry run" for go-live).</span></span> <span data-ttu-id="bd277-114">ただし、何らかの理由でデータベースへのアクセスを必要とする場合は、配置サポート エンジニア (DSE) サービスの要求を通す必要があります。</span><span class="sxs-lookup"><span data-stu-id="bd277-114">However, if you require access to the database for any reason, you must go through deployment support engineer (DSE) service requests.</span></span> <span data-ttu-id="bd277-115">したがって、この方法は効率的ではない場合があります。</span><span class="sxs-lookup"><span data-stu-id="bd277-115">Therefore, this approach might not be efficient.</span></span> <span data-ttu-id="bd277-116">さらに、実稼動環境は、計画された稼動日以前には、長期間使用することはできません。</span><span class="sxs-lookup"><span data-stu-id="bd277-116">Additionally, the production environment isn't available for a long period before the planned go-live date.</span></span>

<span data-ttu-id="bd277-117">UAT は、正式にビルド展開可能なパッケージを配置した後に、実行してください。</span><span class="sxs-lookup"><span data-stu-id="bd277-117">The UAT should be done after you deploy officially built deployable packages.</span></span> <span data-ttu-id="bd277-118">Microsoft Visual Studio で手動でビルドされたパッケージでは実行できません。</span><span class="sxs-lookup"><span data-stu-id="bd277-118">It should not be done on packages that are manually built in Microsoft Visual Studio.</span></span> <span data-ttu-id="bd277-119">理由として、どのコード変更が手動でビルドされたパッケージに含まれていたかを証明する手段が無いからです。</span><span class="sxs-lookup"><span data-stu-id="bd277-119">The reason is that there is no way to prove what code changes were included in a manually built package.</span></span> <span data-ttu-id="bd277-120">公式のビルド システムだけが、特定のビルド内にある正確な変更の保証と監査証跡を提供します。</span><span class="sxs-lookup"><span data-stu-id="bd277-120">Only an official build system provides assurance and an audit trail of the exact changes that are in a specific build.</span></span>

<span data-ttu-id="bd277-121">Retail Modern POS/クラウド POS を使用する場合は、正しいユーザー ロールを使用することを確認してください。</span><span class="sxs-lookup"><span data-stu-id="bd277-121">If you use Retail Modern POS/Cloud POS, make sure that you use the correct user roles.</span></span> <span data-ttu-id="bd277-122">マネージャー、およびより低い権限を持つレジ担当者としてログインすることにより、テストする必要があります。</span><span class="sxs-lookup"><span data-stu-id="bd277-122">You should test by signing in as both a manager and a cashier who has lower privileges.</span></span>

## <a name="performance"></a><span data-ttu-id="bd277-123">パフォーマンス</span><span class="sxs-lookup"><span data-stu-id="bd277-123">Performance</span></span>
### <a name="channel-performance"></a><span data-ttu-id="bd277-124">チャネルの実績</span><span class="sxs-lookup"><span data-stu-id="bd277-124">Channel performance</span></span>

<span data-ttu-id="bd277-125">場合によっては、チャネルのパフォーマンスが期待したほど良くないことがあります。</span><span class="sxs-lookup"><span data-stu-id="bd277-125">In some cases, channel performance might not be as good as you expected.</span></span> <span data-ttu-id="bd277-126">パフォーマンスの低下は、多くの場合次の要因によって発生します。</span><span class="sxs-lookup"><span data-stu-id="bd277-126">Poor performance is often caused by the following factors.</span></span> <span data-ttu-id="bd277-127">このリストは、影響の高い順に並べられています。</span><span class="sxs-lookup"><span data-stu-id="bd277-127">This list is in order from highest to lowest impact.</span></span>

- <span data-ttu-id="bd277-128">Retail サーバーへの追加カスタム呼び出し。</span><span class="sxs-lookup"><span data-stu-id="bd277-128">Additional custom calls to Retail Server.</span></span> <span data-ttu-id="bd277-129">追加の Retail サーバーの呼び出しを使用して製品を拡張すると、多くの場合パフォーマンスが大幅に低下します。</span><span class="sxs-lookup"><span data-stu-id="bd277-129">If you extend the product with additional Retail Server calls, performance often decreases significantly.</span></span> <span data-ttu-id="bd277-130">追加処理の可能性だけでなく、ネットワーク待機時間も考慮する必要があります。</span><span class="sxs-lookup"><span data-stu-id="bd277-130">Not only is there a possibility of additional processing, but the network latency must also be considered.</span></span> <span data-ttu-id="bd277-131">Retail サーバーの追加の呼び出しを回避することをお勧めします。</span><span class="sxs-lookup"><span data-stu-id="bd277-131">We recommend that you try to avoid any additional Retail Server calls.</span></span> <span data-ttu-id="bd277-132">多くの場合、拡張機能のプロパティを使用して、既存の Commerce Runtime (CRT) のハンドラーやトリガーの拡張によって、同じタスクを実行できます。</span><span class="sxs-lookup"><span data-stu-id="bd277-132">Often, you can accomplish the same tasks by using extension properties and extending existing Commerce runtime (CRT) handlers or triggers.</span></span>

### <a name="performance-testing"></a><span data-ttu-id="bd277-144">パフォーマンス テスト</span><span class="sxs-lookup"><span data-stu-id="bd277-144">Performance testing</span></span>

<span data-ttu-id="bd277-145">通常、システムのパフォーマンスをテストするときは、多くの共有リソースが競合するコンポーネントに焦点を絞る必要があります。</span><span class="sxs-lookup"><span data-stu-id="bd277-145">Typically, when you test the performance of a system, you should focus on components where there is competition for many shared resources.</span></span> <span data-ttu-id="bd277-146">これらのリソースは、さまざまなプロジェクト、顧客、または要件ごとに異なる場合があります。</span><span class="sxs-lookup"><span data-stu-id="bd277-146">These resources might differ for different projects, customers, or requirements.</span></span>

<span data-ttu-id="bd277-147">ボトルネックが発生するいくつかの理由は以下の通りです。</span><span class="sxs-lookup"><span data-stu-id="bd277-147">Here are some of the reasons why bottlenecks can occur:</span></span>

- <span data-ttu-id="bd277-148">Finance and Operations でのリソース集中型の計算は、明細書の転記、チャネル データ同期の計算変更など、大規模な製品品揃えを含む倉庫管理の操作、および MRP (Material Requirements Planning) の実行</span><span class="sxs-lookup"><span data-stu-id="bd277-148">Resource-intensive calculations in Finance and Operations, such as statement posting, change calculations for channel data synchronization, warehousing operations that involve a large product assortment, and MRP (Material Requirements Planning) runs</span></span>
- <span data-ttu-id="bd277-149">いくつかの Retail サーバー (クラウド内または Retail Store Scale Unit のいずれか) で実行する、複数の端末またはストア用の複雑な小売ビジネス ロジック</span><span class="sxs-lookup"><span data-stu-id="bd277-149">Complex retail business logic for multiple terminals or stores that run on a few Retail Servers (either in the cloud or in a Retail store scale unit)</span></span>
- <span data-ttu-id="bd277-150">統合されたサード パーティ システム (Finance and Operations または Retail サーバーのいずれかから統合)</span><span class="sxs-lookup"><span data-stu-id="bd277-150">Integrated third-party systems (integrated from either Finance and Operations or Retail Server)</span></span>
- <span data-ttu-id="bd277-151">Retail サーバーから頻繁に呼び出されるリアル タイム トランザクション サービス。</span><span class="sxs-lookup"><span data-stu-id="bd277-151">Real-time transaction services that are frequently called from Retail Server.</span></span>
- <span data-ttu-id="bd277-152">非標準または拡張された標準機能 (たとえば、カスタム WHS コードを使用する拡張明細書転記)</span><span class="sxs-lookup"><span data-stu-id="bd277-152">Non-standard or extended standard functionality (for example, extended statement posting that uses a custom WHS code)</span></span>

<span data-ttu-id="bd277-153">一般的に、既定および非リアルタイム POS 操作は、専用のリソース (POS がインストールまたは実行されているコンピューター) を持っているため、ボトルネックとはみなされません。</span><span class="sxs-lookup"><span data-stu-id="bd277-153">In general, default and non-real-time POS operations aren't considered bottlenecks because they have their own dedicated resource: the computer that the POS is installed or running on.</span></span> <span data-ttu-id="bd277-154">パフォーマンスの問題は、通常、ビジネス ロジックまたは Retail サーバーへの「Chatty」呼び出しによって発生します。</span><span class="sxs-lookup"><span data-stu-id="bd277-154">Performance issues are typically caused by the business logic or "chatty" calls to Retail Server.</span></span>

<span data-ttu-id="bd277-155">パフォーマンス テストはこのトピックで前述した情報を使用して、既にいくつかの初期最適化を完了した後に行う方が理想的です。</span><span class="sxs-lookup"><span data-stu-id="bd277-155">Ideally, performance testing should be done after some initial optimizations have already been completed by using the information earlier in this topic.</span></span> <span data-ttu-id="bd277-156">単一のユーザーまたはプロセスでシステムが正常に動作しない場合、同時ユーザーまたはプロセスでは正常に動作しません。</span><span class="sxs-lookup"><span data-stu-id="bd277-156">If the system doesn't perform well for a single user or process, it won't perform well for concurrent users or processes.</span></span> <span data-ttu-id="bd277-157">詳細については、[Retail Channel Performance](https://dynamicsnotes.com/retail-channel-performance-investigations/) を参照してください。</span><span class="sxs-lookup"><span data-stu-id="bd277-157">For more information, see [Retail Channel performance investigations](https://dynamicsnotes.com/retail-channel-performance-investigations/).</span></span> <span data-ttu-id="bd277-158">さらに、Finance and Operations ドキュメントで、「PerfSdk」または「Trace parser」を検索します。</span><span class="sxs-lookup"><span data-stu-id="bd277-158">Additionally, in the Finance and Operations documentation, search for "PerfSdk" or "Trace parser."</span></span>

<span data-ttu-id="bd277-159">すべてのプロジェクトは異なっているため、実行する必要のある正確なパフォーマンス テストについての一般的な回答は困難です。</span><span class="sxs-lookup"><span data-stu-id="bd277-159">Because every project is different, it's difficult to give a general answer about the exact performance tests that must be run.</span></span> <span data-ttu-id="bd277-160">たとえば、トランザクションの販売明細行が少ない (すべての店舗で 1 日あたり 10 万未満) 場合、および明細書転記にカスタム拡張コードが追加されていない場合は、転記にパフォーマンス テストを行う必要はありません。</span><span class="sxs-lookup"><span data-stu-id="bd277-160">For example, if the count of transaction sales lines is low (less than 100,000 per day for all stores), and if no custom extension code has been added for statement posting, a performance test should not be required for posting.</span></span> <span data-ttu-id="bd277-161">ただし、販売明細行が大幅に高い場合、または主要なカスタム変更が加えられた場合は、パフォーマンス テストを転記することをお勧めします。</span><span class="sxs-lookup"><span data-stu-id="bd277-161">However, if the count of sales lines is substantially higher, or if major custom changes have been added, a performance test for posting is a good idea.</span></span>

<span data-ttu-id="bd277-162">通常、すべての環境でハードウェア機能は異なります。</span><span class="sxs-lookup"><span data-stu-id="bd277-162">Usually, the hardware capabilities of every environment differ.</span></span> <span data-ttu-id="bd277-163">ただし、コードおよびデータが類似している場合、通常他の環境でもパフォーマンスの問題は起こり得ます。</span><span class="sxs-lookup"><span data-stu-id="bd277-163">However, performance issues can usually be reproduced in other environments if the code and data are similar.</span></span> <span data-ttu-id="bd277-164">パフォーマンス テストに実稼働環境を使用することをお勧めしません。</span><span class="sxs-lookup"><span data-stu-id="bd277-164">We don't recommend that you use a production environment for performance testing.</span></span> <span data-ttu-id="bd277-165">開発、テスト、および運用環境で同じデータを使用することをお勧めします。</span><span class="sxs-lookup"><span data-stu-id="bd277-165">A good practice is to use the same data in development, test, and production environments.</span></span> <span data-ttu-id="bd277-166">開発環境は、修正プログラムの作成と検証に使用されます。</span><span class="sxs-lookup"><span data-stu-id="bd277-166">The development environment can then be used to work on and verify a fix.</span></span> <span data-ttu-id="bd277-167">多数のパフォーマンス クリティカル コード パスはデータ依存のため、 Contoso サンプル データベースでは同じ問題が発生しないことがあります。</span><span class="sxs-lookup"><span data-stu-id="bd277-167">Because many performance-critical code paths are data-dependent, the same issues might not be seen for a Contoso sample database.</span></span>
